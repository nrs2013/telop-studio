# 🚨 データ保護ルール — 絶対に守ること

**作成のきっかけ：**
2026-04-25、Claude が譜割タブ（localStorage `telop-score-v2-*`）の **マイグレーション** を勝手に実行し、の む さんが入力した歌詞データを **永久に破壊** した。これは商業ツールではあってはならない事故。

---

## 守るべき 3 ルール

### ルール 1：既存データを変換するコードは書かない（最優先）
- localStorage / IndexedDB / project state を **読み込む時** はそのままコピーする
- 形を変える処理（split, filter, map で構造を変える等）は **禁止**
- とくに **書き戻し** が伴う場合は絶対 NG（破壊的変更が永続化される）

### ルール 2：どうしても変換が必要な時は、必ず先にバックアップ
- 元データを **別キー** に保存：`${元キー}-backup-${ISO日時}`
- バックアップが完了してから初めて加工
- ユーザーに **事前承認**：「こう変換するけど、戻せなくなるけど OK？」
- 承認なしで実行しない

### ルール 3：データ形式を変えたい時は新キーで並行運用
- 古いキー（v2）には絶対に書き込まない、上書きしない
- 新キー（v3）で空状態から再スタート
- 古いキーは半永久的に残し、ユーザーがいつでも参照できるようにする

---

## 今回の事故の経緯（教訓として残す）

1. 過去の入力データ：scoreRows[0].section = "INTRO\n\n1F\n\n1G\n..." と改行で詰まってた（textarea で改行入力できた結果）
2. Claude が「これを行展開しよう」とマイグレーションを書く（ユーザー承認なし）
3. lyric を空行で分割して section/bars 行に割り当てるロジックを実装
4. 割り当てロジックがズレて、ほとんどの行に空 lyric が入る
5. その状態が v2 キーに自動保存（useEffect で）→ **元データの歌詞が永久に消える**
6. 気付いた時には localStorage に残ってるのは破壊済みのデータのみ

**何が悪かったか：**
- データを書き換えるコードを **ユーザー確認なし** に実装した
- バックアップを取らずに変換した
- 「LYRIC を空行で分割」というロジックの **正しさを保証できない** のに本番に投入した
- v2 キーに **書き戻し** たため、元データが永久に上書きされた

## 実装上のチェックリスト（コードレビュー時必須）

新しいコードを書く前・push する前に、以下を確認：

- [ ] **`localStorage.setItem()` で既存キーを上書きしてないか？**（新規キーならOK、既存キー上書きは要確認）
- [ ] **読み込み時にデータ加工してないか？**（`.split()`, `.filter()`, `.map()` で形を変えていないか）
- [ ] **JSON.parse 後にプロパティを書き換えてないか？**
- [ ] **マイグレーションコードを書いていないか？**（書いていれば、別キーへの並行運用に切替）
- [ ] **新機能を入れる時、既存データに影響しないか？**

- 譜割タブのデータは **絶対に勝手に変換しない**
- 「形を整える」と言って実行するコードを **書かない**

- バックアップなしで操作しない
